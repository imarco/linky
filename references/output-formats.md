# 输出格式指南

用户可以选择四种输出方式。默认方式可通过 `~/.config/linky/config.toml` 的 `default_output` 字段配置。

输出方式只决定报告投递到哪里或以什么载体交付；输出风格决定报告怎么写。输出风格由 `output_style` 字段配置，详细规范见 `references/output-style-guide.md`。

所有输出格式都应从结构化 `ReportData` 渲染，而不是各自重新解析 Markdown。这样可以保证 Markdown、Notion、Obsidian 和 Prompt 模式看到同一组字段：名称、URL、类型、访问状态、一句话结论、我的判断、建议行动、trace 摘要和研究价值。

---

## 1. Notion 输出

使用 Notion MCP 工具将研究结果写入 Notion。

### 创建方式

1. 先在用户指定的 Notion 页面下创建一个新页面，标题格式：`链接研究报告 - {日期} - {主题关键词}`
2. 页面内容按报告结构（A/B/C）组织
3. 如果用户有指定的 Notion 数据库，也可以创建数据库条目：
   - 每个链接一行
   - 字段：名称、URL、类型、访问状态、一句话结论、建议行动、研究价值评级

### 注意事项

- 使用 `notion-search` 先确认目标位置存在
- 长内容分段写入，Notion 有块大小限制
- 保留原始链接为可点击的 URL

---

## 2. 单文件 Markdown

输出一个完整的 `.md` 文件。

### 文件命名

`link-research-{YYYY-MM-DD}-{主题关键词}.md`

### 文件位置

- 默认：当前工作目录
- 可通过 `config.toml` 的 `output_dir` 配置
- 用户也可以在调用时指定

### 内容

完整的 A + B + C 报告，使用标准 Markdown 格式。

---

## 3. 多文件 Markdown

按类型分文件夹，每个链接一个文件。

### 目录结构

```
link-research-{YYYY-MM-DD}-{主题关键词}/
├── README.md               # 研究总览(A) + 研究结论(C)
├── git-repos/
│   ├── 01-{项目名}.md
│   └── 02-{项目名}.md
├── products/
│   ├── 01-{产品名}.md
│   └── 02-{产品名}.md
├── docs-tutorials/
│   └── 01-{资源名}.md
├── articles/
│   ├── 01-{文章标题}.md
│   └── 02-{文章标题}.md
├── platforms/
│   └── 01-{平台名}.md
├── social-media/
│   └── 01-{帖文主题}.md
└── others/
    └── 01-{名称}.md
```

### 单个文件内容

每个文件只包含该链接的完整分析卡，文件开头加上 YAML frontmatter：

```yaml
---
name: "{名称}"
url: "{原始链接}"
type: "{类型}"
access_status: "{访问状态}"
one_line: "{一句话结论}"
research_value: {1-5}
date_analyzed: "{YYYY-MM-DD}"
tags: ["{标签1}", "{标签2}"]
---
```

## 4. Prompt 模式（省钱模式）

核心思路：用 Claude 完成高价值的正文提取、分类和结构化，然后生成精简的 prompt 让更便宜的 AI 平台完成后续任务（翻译、改写、二次整理、生成衍生内容等）。

### 何时使用

- 需要基于研究结果做进一步加工（翻译、扩写、生成选题等）
- 后续任务不需要 Claude 级别的能力
- 想节省 Claude token

### Prompt 结构

生成一个通用的、结构良好的 prompt，不做特定平台适配，依赖目标平台自身能力：

```markdown
# 任务说明

我已经对 {N} 个链接进行了研究和分类。以下是结构化的研究数据。
请基于这些数据，帮我{用户指定的任务}。

## 研究数据

{嵌入精简的分析卡摘要，保留关键结构化字段}

## 研究结论

{C 部分的关键结论}

## 具体要求

{根据用户意图生成具体指令}
```

### 设计原则

- **通用性优先**：不针对任何特定平台做适配，生成的 prompt 应该在任何 AI 平台上都能正常工作
- **精简但完整**：包含足够的结构化数据支撑后续任务，但去掉冗余信息，保持 prompt 紧凑
- **任务导向**：末尾的"具体要求"部分是核心，要根据用户的实际意图生成清晰、可执行的指令
- 不要包含"你是研究分析师"之类的角色设定——那是本 skill 的角色，下游任务应该根据实际需求设定角色
- 如果内容太长，自动拆分为多个 prompt，每个开头注明"第 X/Y 部分"
- 附带简短使用说明：告诉用户粘贴顺序

## 输出方式的选择逻辑

1. 如果 `config.toml` 中有 `default_output`，使用配置的默认值
2. 否则，首次使用时询问用户偏好，并建议保存为默认值
3. 用户可以在任何时候通过自然语言切换输出方式（如"这次输出到 Notion"）
4. 如果用户没有明确指定且没有配置，默认使用**单文件 Markdown**

## 输出风格的选择逻辑

1. 如果用户本次明确指定风格（如"用极简白话"），优先使用本次指定
2. 否则读取 `config.toml` 中的 `output_style`
3. 如果未配置，默认使用 `standard`
4. 输出风格不改变输出方式；例如 `default_output = "obsidian"` 且 `output_style = "极简白话"` 表示把极简白话版本写入 Obsidian
